# 同期補正の根本的問題（2026-06-30 分析）

これまでの drift_measure → ppm補正 → PLL補正 の流れを再検証した結果，
**スレーブ間同期はすでに要件を満たしており，ppm/pll 補正の有無で聴感は変わらない**ことが判明した．

---

## 1. drift_measure が測っている値

```
drift_us = 「自分がSYNCを受け取った間隔」−「公称小節長」
```

つまり **master→自分 の1対1の配信ジッタ**であり，各slaveが独立に自分のことしか見ていない．

### スレーブ間の同期精度は分からない

スレーブ間の誤差（聴感上のズレ）とは「fluteとclarinetが**同じ**小節頭を**どれだけ同時に**受け取れたか」であり，slave同士の受信タイミングの**差分**である．

極端な例：
- 全slaveの drift_us が毎回 +3000µs → drift_measure 上は「3msもズレてる！」→ 実際は全員同じだけ遅れているので**完全に同期している**
- 各slaveの drift_us が ±500µs だがバラバラな方向 → 数値は小さいが聴感上はズレている

**drift_measure の値を減らす方向に最適化しても，スレーブ間同期の改善に直結しない．**

### 3ms 誤差の内訳

drift_measure で観測される 3ms 程度の誤差の内訳は以下のとおりである．

| 原因 | 寄与 | 性質 |
|------|------|------|
| masterの `analogRead()` + float演算の揺らぎ | 数百µs | ランダム |
| I2C 4台逐次送信のタイミング変動（400kHz・BPM変化時） | 〜0.4ms | 準系統的 |
| ISR内 `micros()` のジッタ | 〜1ms | ランダム（ノイズ床） |

すべて **master側の送信揺らぎ + 測定器自身のノイズ**であり，slave側の補正では原理的に消せない．

---

## 2. PPM補正が効かない理由

PPM補正は「slaveの水晶のクセで，小節内の拍タイミングがズレる分を補正する」ものである．

### 毎小節SYNCリセットだから効く区間が短すぎる

slave は毎小節，SYNC受信時刻で `barStartUs` をリセットしている：

```c
barStartUs = t;  // SYNC受信時刻で毎回上書き
```

PPM補正が効く区間は「SYNC受信（beat 0）から小節末（beat 3）まで」の最大1小節（BPM120 で 2秒間）である．各拍は slave 側で `beatUs = barLengthUs / 4` から発火するため，PPM補正はbeat 1〜3 の発火時刻を**実際に動かす**．

beat 3 は小節の 3/4 地点（BPM120で1.5秒）にあるため，各楽器の絶対ズレは以下のとおりである．

| 楽器 | ppm | beat 3 の絶対ズレ（実時間） |
|------|-----|---------------------------|
| drum（1872ppm） | +1872 | −2,808µs（最早） |
| flute（1419ppm） | +1419 | −2,129µs |
| clarinet（618ppm） | +618 | −927µs |
| organ（−584ppm） | −584 | +876µs（最遅） |

楽器間の相対ズレ（主要ペア）:

| ペア | 相対ズレ | 聴覚閾値との関係 |
|------|---------|----------------|
| flute − drum | 約 679µs | 大きく下回る |
| clarinet − organ | 約 1,803µs | 下回る |
| clarinet − drum | 約 1,881µs | 下回る |
| flute − organ | 約 3,005µs | **範囲内（3〜5ms）** |
| **organ − drum（最大）** | **約 3,684µs** | **範囲内（3〜5ms）** |

> **注**: 初版ではflute−drum間の678µsのみを比較して「聴覚閾値を大きく下回る」と
> 結論していたが，ドリフト方向が逆のorgan（−584ppm）を含めると最大ペア差は
> 約3.7msとなり，聴覚閾値（3〜5ms）の範囲内に入る．

MOP1（30ms）には大幅に余裕があるため**同期要件自体は満たされている**．
ただし「聴感に一切影響しない」とは断定できない．実機で耳で確認した上で，
知覚できるズレがあれば対策を考える余地がある．

→ **PPM補正は beat 1〜3 を動かしてはいるが，MOP1 基準では不要．
聴覚閾値基準では最大ペア（organ−drum）が境界付近にあり，実聴確認で判断する．**

### BPM可変で前提が崩れる

PPM値は特定のBPMで計測した固定値であり，BPMがポテンショメータで変わると計測時と異なる条件で固定補正をかけることになる．

---

## 3. PLL補正が効かない理由（かつ有害な理由）

PLL は `barLengthUs`（= 小節長 = 拍間隔の基準）を調整している．

```c
barLengthUs = calcBarUsCorrected(bpm) + pllCorrectionUs;
```

「前回SYNCからの実測間隔」と「予測間隔」のズレから，次の小節の `barLengthUs` を微調整する仕組みである．

### なぜ無意味か

1. beat 0 は SYNC受信時刻にアンカーされるので，PLLの補正は**関係ない**．
2. beat 1〜3 の発火タイミングへの影響は，PPMと同じ理由で**聴感上無視できるレベル**．
3. 次のSYNCが来たら `barStartUs = t` で**全部リセット**される．

**予測しようが，SYNC受信した瞬間に上書きされるので，予測値を使って先行する処理がない．**

### BPM可変で有害になる理由

PLLは「前の小節の残差」から「次の小節の補正量」を決めている．BPMがポテンショメータで毎小節変わりうる環境では：

- 前の小節の残差は「前のBPMでの残差」
- 次の小節は別のBPM
- **BPM変化分をドリフトと誤認して逆方向に補正をかける可能性がある**

---

## 4. I2C逐次配信スキュー

master の `sendSyncToAll()` が I2C で 4台に逐次送信している：

```c
void sendSyncToAll(uint16_t song_bar, uint16_t bpm_x10) {
  for (byte i = 0; i < NUM_SLAVES; i++) {
    sendSync(slaveADRs[i], song_bar, bpm_x10);  // 400kHz で 1台あたり 約0.14ms
  }
}
```

1台目（flute, 0x10）と4台目（drum, 0x13）の間に **約0.4msの確定的なオフセット**がある（本番は `Wire.setClock(400000)` で 400kHz）．これは毎回同じ方向に同じ量だけズレる**系統誤差**であり，PLLやPPM補正（ランダムなクロックドリフトを補正する技術）では原理的に消せない．ただし 0.4ms は聴覚閾値を下回るため，**この系統オフセットも実害はない**．

---

## 5. 結論

- **drift_measure の値**: スレーブ間同期の指標として不適切（master→自分の1対1ジッタしか見ていない）．
- **PPM補正**: beat 1〜3 を実際に動かす．楽器間ズレは beat 3 で最小ペア（flute−drum）が約 679µs，最大ペア（organ−drum）が約 3.7ms．MOP1（30ms）には大幅に余裕があるが，最大ペアは聴覚閾値（3〜5ms）の範囲内であり，実聴で知覚できるか要確認．
- **PLL補正**: 同様に楽器間ズレへの寄与は小さい．加えて BPM可変では前小節の残差を誤補正するリスクがある．
- **I2C逐次配信スキュー**: 400kHz で約 0.4ms．これも聴覚閾値を下回る．

要件（MOP1 = 30ms）に対して，最大 3.7ms でも桁違いに小さく，**MOP1 は満たされている**．ただし聴覚閾値（3〜5ms）に対しては，最大ペア（organ−drum）の約 3.7ms が範囲内に入るため，「補正なしで聴感に一切影響しない」とは断定できない．

つまり**素の sing_bpm + 400kHz の時点で MOP1 要件は満たしている**が，聴覚閾値基準では organ−drum 間が境界付近にある．

肝心な判断基準は「数字をこれ以上小さくすること」ではなく「実機で耳で聞いてズレが分かるか」である．分からなければ ppm/pll は不要．分かった場合は何らかの補正を検討．それでも不足なら配信方式（GPIO同期ピン等）を検討する．
